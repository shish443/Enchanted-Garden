# Enchanted Garden - Управление Магическим Садом
Сухое тестовое задание на разработку API оргструктуры. Сделал в виде Магического Сада. Чтобы не умереть со скуки, пока писал код. Вместо банальных департаментов и подразделений - Ветки (Branch). Вместо сотрудников - Флора (Flora).
Проект на Go. Аккуратно разложен по слоям. Всё модульно.
## Быстрый запуск
Всё упаковано в Docker. Разворачивается одной командой
1. Склонируй репозиторий:
```bash
git clone https://github.com/shish443/Enchanted-Garden.git
cd Enchanted-Garden 
```
2. Запусти контейнеры:
```bash
docker compose up --build -d
```
Приложение само дождется готовности PostgreSQL. Автоматически накатит миграции через goose. Поднимет сервер на порту :8080.
## Что под капотом (Фичи и гордость)
* **GORM в роли переводчика.** Если на пальцах: как маг и огонь. Маг понимает только заклинания, огонь - команды природы. GORM тут выступает толмачом. Превращает наши Go-структуры в волю базы данных и обратно.
* **Кастомные даты** Настоящая боль. Изначально было пофиг, пока тесты не сломали мою уверенность. Оказывается, стандартное время тянет за собой кракозябру вида T00:00:00Z. Всё падало. Написал кастомный тип DateOnly со своими методами Marshal/UnmarshalJSON. Теперь даты уходят и приходят строго в человеческом формате YYYY-MM-DD. Работает железно.
* **Защита от древесных аномалий.** Ветки умеют рекурсивно вкладываться друг в друга. До 5 уровней в глубину. Сервис жестко следит, чтобы ветка не стала родителем самой себе. Дерево не зациклится. При попытке создать петлю - лови честный 409 Conflict.
* **Умное удаление.** Реализовано каскадное удаление на уровне БД. Стираешь ветку - подчищается вся флора на ней. Также есть режим reassign. Жителей можно оперативно переселить в другое подразделение перед удалением текущего.
* **Graceful Shutdown (Сад не завянет).** Прикрутил плавное завершение. При Ctrl+C приложение не падает с грохотом, а дожидается окончания текущих HTTP-запросов, закрывает пул соединений с Postgres. Аккуратно тушит свет. База скажет спасибо.
* **Красивые структурированные логи.** Подключен стандартный slog. Никакого хаоса в консоли. Только чистый, красивый JSON. 
* **Тесты.** Написал и юнит-тесты логики на testify, и HTTP-тесты хендлеров, и даже интеграционный branch_db_test.go. На своей шкуре прочувствовал: кривые тесты - кривые результаты.

## Структура проекта
Код разложен по полочкам:
* cmd/main.go - точка входа, инициализация зависимостей, запуск.
* db/migrations/ - SQL-скрипты миграций для goose.
* internal/config/ - чтение конфига напрямую из окружения докера.
* internal/handler/ - HTTP-ручки и роутер.
* internal/service/ - чистая бизнес-логика и валидация.
* internal/repository/ - запросы к PostgreSQL через репозиторий.
* internal/model/ - GORM-структуры данных и DTO.
* logger/ - конфигурация логирования slog.
